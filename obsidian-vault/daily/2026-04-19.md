## Решения

- **Вариант А — рубим миграции по одной** вместо одного общего SQL-файла. Причина: каждая миграция изолирована, можно катить и проверять независимо. 9 штук — не так уж много.
- **Старая `contacts` (каналы связи лида) → `lead_channels`**. Новая `public.contacts` теперь действительно «телефонная книга партнёра» как и задумано в CLAUDE.md. FK переименован корректно.
- **`templates`: расширение вместо пересоздания** — дропнул узкий check на `type`, оставил колонку nullable для обратной совместимости. Новое поле `category` стало primary.
- **`followup_delay_hours` nullable** = off. Не стал использовать -1/0 как sentinels.
- **Системные шаблоны = `partner_id IS NULL`.** Видны всем authenticated. Свои — только владельцу. INSERT только в свои.
- **`training_modules`/`lessons` read-only для всех authenticated.** Админский write — потом, через service_role.
- **`events_log` INSERT оставлен открытым для anon** — лендинг должен писать события без авторизации. Защита на стороне формы (rate-limit, CAPTCHA — отложено).
- **`partners` SELECT открыт всем authenticated** — для структуры, ref-валидации, team display. Поля (name, email, phone, ref_code) считаются не чувствительными в MLM-контексте.
- **Все mock-данные намеренно захардкожены в JS каждой вкладки** — не в БД. Переключение на реальные `supabase-js` запросы — в следующей сессии.
