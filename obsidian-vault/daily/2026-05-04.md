## Решения

- **Phase B path = A (processed_at колонка)** — минимальное изменение, race-window узкий при cron ≥ 2 мин, explicit semantics. Альтернативы B (task_id FK) и C (RPC atomic) перебросили в backlog.
- **Score gate = no gate** — все succeeded qualifications порождают ai_recommendations и tasks. Сайра решает на UI. Если потом захотим фильтр — добавить `confidence=gte.0.6` в WF4 fetch.
- **WF4 schedule 10 мин → 2 мин** для smoke parity с WF3. Можно вернуть 10 мин — task creation не критично по латенси.

## Заметки

- **Side-effect partial-failure pattern**: n8n не делает rollback HTTP-ов между нодами. Если node 30 succeed но 50 fail — execution status=error, но 30 уже committed. Acceptable для idempotent flows (повторный tick найдёт processed_at и пропустит). Нужно учитывать в дизайне.
- **`ai_jobs.updated_at` semantic**: `claim_next_ai_job` RPC бампит updated_at при start (running). Node 70 PATCH `{status:'succeeded', result:{...}}` без `updated_at` колонки — Supabase колонку не трогает. Итог: `updated_at` показывает **start**, не finish. Если хотим finish-time — добавить `updated_at: now()` в node 70 jsonBody.
- **events_log.actor whitelist**: только `me|ai|system|partner|lead|anon`. Кастомные значения отвергаются CHECK. Для cron/n8n использовать `'system'`.
- **partial index pattern**: `WHERE processed_at IS NULL` сильно сужает hot path WF4 GET — будущее масштабирование сохранит latency.
