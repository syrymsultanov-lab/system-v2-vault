### Найдены 2 капкана n8n (зафиксированы в memory)

**1. n8n jsonBody без `=` префикса = literal text.**
- `"jsonBody": "{...{{ expr }}...}"` — `{{ }}` остаётся как текст, n8n parser отдаёт `"JSON parameter needs to be valid JSON"`
- `"jsonBody": "={...{{ expr }}...}"` — `=` активирует expression mode, `{{ }}` интерполируется
- Memory: `feedback_n8n_jsonbody_expression.md`

**2. Nested `{}` внутри `{{ }}` ломают expression parser.**
- Работает: `={{ {status:'running'} }}` (1 уровень)
- Ломается: `={{ {status:'x', result:{error:'y'}} }}` → "invalid syntax" в renderExpression
- Причина: JS parser ambiguity — `{a:1}` в expression position парсится как блок с labeled statement
- Решение: literal JSON template + `{{ JSON.stringify(...) }}` для значений

### Реальный root cause OpenAI 401 (это НЕ был corrupted env)

Изначально решил, что вчерашний `IV4A`-ключ был перезаписан/протух — на проде WF3 слал `...PncA`, контейнер env правильный (длина 164, хвост IV4A). Парадокс.

**Истинная причина:** мой собственный deploy-скрипт `scripts/upsert_wf.py` (унаследованный от `deploy_wf_from_json.py`) запускал `deep_resolve()` — сканировал JSON и заменял `={{ $env.X }}` на значение `env["X"]` из ЛОКАЛЬНОГО `.env` корня репо. Локально у меня лежал старый PncA-ключ. После PUT — workflow на проде имел хардкод `Bearer sk-proj-...PncA`, контейнерный env вообще не читался.

Подтверждено через GET `/workflows/{id}`: до фикса Authorization header был `={{'Bearer ' + 'sk-p...3QePncA'}}`. После выпиливания deep_resolve — `={{'Bearer ' + $env.OPENAI_API_KEY}}`. WF3 заработал с первого прогона после фикса.

**Memory:** `feedback_n8n_deploy_no_inline_env.md`

## Заметки

- **Симптомы 401 от OpenAI бывают трёх причин с одинаковым выхлопом:** (а) ключ невалиден/отозван (вчерашний случай), (б) корректный ключ записан с обрезкой через Browser Terminal (вчерашний случай 1), (в) deploy-pipeline хардкодит другой ключ из локального .env (сегодняшний случай). Различить можно только сравнением хвостов: env в контейнере vs actual header в deployed workflow body.
- **Способ диагностики хардкода в n8n workflow:** `GET /api/v1/workflows/{id}` → найти headerParameters → сравнить значение с `={{ $env.X }}` шаблоном. Если видно raw token — есть хардкод.
- **n8n REST API позволяет полный CRUD workflows без Browser Terminal/UI.** Endpoints: `GET /api/v1/workflows`, `POST /api/v1/workflows`, `PUT /api/v1/workflows/{id}`, `POST /api/v1/workflows/{id}/activate|deactivate`, `GET /api/v1/executions?workflowId=X&includeData=true`. Auth через `X-N8N-API-KEY` header.
- **n8n executions API → полная диагностика runtime ошибок:** `GET /executions/{id}?includeData=true` отдаёт `data.resultData.runData[node][run].error` и `.data.main` с исходниками output. Позволяет видеть rendered jsonBody, error message, stack без захода в UI.
- **PostgreSQL FOR UPDATE SKIP LOCKED уже работал в `claim_next_ai_job` с самого начала** — миграция от 2026-04 (точная дата неизвестна). Проблема была в WF3, который не вызывал RPC, а делал raw SELECT+PATCH без блокировок.
