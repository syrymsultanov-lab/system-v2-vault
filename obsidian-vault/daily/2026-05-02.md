### Что чинили (две независимые проблемы, обе невидимые)

**1. SUPABASE_SERVICE_ROLE_KEY был обрезан в `.env`.**
В n8n execution для Fetch Queued Jobs увидел в headers: `apikey: eyJ...iat":17752` — JWT обрывается посередине payload, без подписи. Та же обрезанная строка лежала и в `docker exec n8n-n8n-1 env` (вчера 2026-05-01 не заметил, потому что не сравнивал длину). Supabase отвечал 401 "Invalid API key" — звучит как неверный ключ, по факту ключ просто незаконченный.

**Причина:** при записи длинной строки JWT через Browser Terminal (даже коротким `echo >>`) часть символов теряется. Точную точку обрыва не нашёл, но оба раза (Supabase и OpenAI ниже) симптом одинаковый.

**Фикс:** новый подход через base64.
- Локально кодируем секрет: `printf '%s' 'KEY' | base64 -w0`
- Режем base64 на 2 части по ~140 символов (если >150 — Browser Terminal может опять разорвать)
- `echo -n 'PART1' > /tmp/k.b64`, `echo -n 'PART2' >> /tmp/k.b64`
- `echo "KEY=$(base64 -d /tmp/k.b64)" >> /docker/n8n/.env && rm /tmp/k.b64`
- Проверка: `grep '^KEY=' .env | wc -c` должно совпадать с `len("KEY=") + len(secret) + 1`. То же после `docker compose up -d` через `docker exec ... env | grep ...`.

Записано в `feedback_browser_terminal_paste.md`.

**2. OPENAI_API_KEY невалиден.**
После фикса Supabase цепочка дошла до OpenAI Qualification → 401 `invalid_api_key` для ключа `sk-proj-...ARZ4`. По длине (156 символов) и по тому, что хвост в env-vars и в OpenAI-ошибке совпадал, ключ долетел до OpenAI целиком — то есть **сам ключ был отозван/невалиден**. ## Решения

- **Длинные секреты → всегда base64 + проверка длины.** Без проверки длины невозможно отличить «обрезано при записи» от «изначально невалиден» — обе проблемы дают одинаковый 401.
- **Перед SQL по незнакомой/малознакомой таблице — сначала `information_schema.columns`.** Daily-ноты с SQL-сниппетами устаревают и врут.
- **WF3 фактически готов** — переезжает с `[~]` на `[x]` в SYSTEM V2.md (с пометкой про `result` и `claim_next_ai_job`).
